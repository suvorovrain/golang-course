# Домашнее задание №5
## Кэширование и Rate Limiting в API Gateway

---

## Контекст

На текущем этапе система уже реализована как набор микросервисов:

- API Gateway
- Processor
- Collector
- Subscribe

API Gateway принимает внешние HTTP-запросы и проксирует их в остальные сервисы по gRPC.

В данном задании необходимо улучшить устойчивость и производительность системы за счёт:

- кэширования ответов,
- ограничения количества запросов (rate limiting).

---

## Задача

Необходимо реализовать в **API Gateway**:

1. **Кэширование запросов с использованием Redis**
2. **Rate Limiter для ограничения входящих запросов**

---

## Что нужно реализовать

### 1. Интеграция Redis

API Gateway должен:

- подключаться к Redis,
- использовать его как хранилище для кэша,
- использовать Redis для хранения состояния rate limiter, если выбран распределённый вариант.

Разрешается использовать любую библиотеку для работы с Redis, например `go-redis`.

---

### 2. Кэширование запросов

Кэшировать ответы для endpoint’ов:

- `GET /api/repositories/info`
- `GET /api/subscriptions/info`

Поведение:

1. Сформировать ключ кэша на основе параметров запроса.
2. Проверить наличие значения в Redis:
    - если найдено — вернуть его (**cache hit**),
    - если нет — выполнить обычную логику (**cache miss**).
3. После получения ответа — сохранить его в Redis.

---

### 3. TTL кэша

TTL должен:

- задаваться в конфигурации в секундах,
- применяться ко всем кэшируемым записям.

Пример конфигурации:

```
cache:
  ttl_seconds: 60
```

---

### 4. Rate Limiting

Необходимо ограничить количество запросов к API Gateway.

Параметры ограничения должны задаваться в конфигурации:

```
rate_limit:
  requests_per_second: 5
  burst: 10
```

Требования:

- ограничение должно применяться минимум на уровне клиента, например по IP;
- допускается глобальный limiter как упрощённый вариант;
- при превышении лимита API должен возвращать `429 Too Many Requests`.

Пример ответа:

```
{
  "error": "rate limit exceeded"
}
```

---

### 5. Порядок обработки запроса

Запрос должен обрабатываться в следующем порядке:

1. Проверка rate limit.
2. Проверка кэша.
3. Выполнение бизнес-логики, если произошёл cache miss.
4. Сохранение результата в кэш.

---

### 6. Обработка отказов

Система должна корректно работать при недоступности Redis:

- кэш не используется;
- запросы продолжают обрабатываться через обычную логику;
- rate limiter может деградировать до in-memory варианта.

При превышении лимита запрос не должен доходить до downstream сервисов.

---

### 7. Конфигурация

Добавить в конфиг API Gateway параметры Redis, кэша и rate limiter.

Пример:

```
redis:
  host: localhost
  port: 6379

cache:
  ttl_seconds: 60

rate_limit:
  requests_per_second: 5
  burst: 10
```

---

## Требования

- Все изменения реализуются только в **API Gateway**.
- Остальные сервисы (**Processor, Collector, Subscribe**) не изменяются.
- Взаимодействие между сервисами остаётся прежним через gRPC.
- Сохранить Clean Architecture:
    - **Domain**
    - **Use Cases**
    - **Controller/Handler**
    - **Adapter**
- Не ломать существующие API-контракты.
- Корректно обрабатывать ошибки:
    - недоступность Redis,
    - превышение rate limit,
    - ошибки сериализации и десериализации кэша.
- Написать тест для rate limiter
  - Можно в виде bash-скрипта с использованием сторонних утилит (напр. `vegeta`)

---

## Docker и окружение

В `docker-compose.yaml` необходимо добавить:

- Redis,
- обновлённый API Gateway.

Все сервисы должны запускаться одной командой.

---

## Ожидаемое поведение

### Cache hit

Повторный запрос возвращается из Redis без обращения к downstream сервисам.

### Cache miss

Данные получаются из системы и сохраняются в кэш.

### Rate limit exceeded

При превышении лимита возвращается `429 Too Many Requests`.

Downstream сервисы не вызываются.
